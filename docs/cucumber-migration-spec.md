# Cucumber 受け入れテスト移行仕様

策定日: 2026-07-14
対象: Issue [#110](https://github.com/yasuhito/deadloop/issues/110)（親課題 [#102](https://github.com/yasuhito/deadloop/issues/102)）

## 1. 目的

この仕様は、deadloop の公開仕様、利用者から見える状態遷移、安全契約を、実行可能な Cucumber Markdown（MDG）へ段階的に移すための実装契約である。実装担当はこの仕様を基に作業を Issue と PR に分割し、追加の製品方針判断をせずに移行を進める。

移行後は `.feature.md` を受け入れ仕様の唯一の正本とする。仕様の説明と実行可能な例を同じファイルに置き、README や別の設計文書には内容を転載せず正本へのリンクだけを置く。

### 範囲

- Cucumber の実行基盤、記述規約の自動検査、CI への組込み
- [既存テスト分類](cucumber-test-classification.md)で Cucumber 候補とした保証の移行
- 移行時の同等性確認と、完全に置換できた Vitest の削除
- 日本語での全面移行と実運用確認
- 明示承認後の英語への一括移行

### 範囲外

- 小さな純粋関数、引数変換、パーサー、実行基盤アダプター、静的ファイル・配布物・プロンプト検査など、Vitest 継続と分類したテストの Cucumber 化
- Cucumber から別形式の仕様書やレポートを生成する仕組み
- 説明文を Cucumber Messages、JSON、AST から回収または再生成する仕組み
- Cucumber 移行と同時に製品動作を変更すること
- 日本語版と英語版を併存させること

## 2. 採用構成

### 実行環境

次の組合せを採用する。

- CI の Node.js 22
- Cucumber-JS 13 系
- 現行の CommonJS パッケージ構成
- `tsx/cjs` を登録して実行する TypeScript ステップ定義
- 現行の TypeScript 6 系

`@cucumber/cucumber` と `tsx` は開発依存に追加し、導入 PR で選んだ正確な版を範囲指定なしで `package.json` と `package-lock.json` に固定する。版更新時には、日本語 MDG と TypeScript ステップ定義を端から端まで実行する回帰確認を必須とする。

### Cucumber 設定

`cucumber.cjs` を唯一の Cucumber 設定入口にし、次を明示する。

- 実行対象を `.feature.md` に限定した探索先
- TypeScript のステップ定義とサポートコードの探索先
- `requireModule: ['tsx/cjs']`
- 日本語運用中の `language: 'ja'`
- `strict: true`

設定と `package.json` の `scripts` に重複した Cucumber オプションを置かない。未定義、曖昧、pending、構文解析失敗、assertion 失敗はいずれも非 0 終了にする。失敗したテストは自動再実行しない。

各シナリオには独立した World を作り、シナリオ間で共有する可変状態を持たせない。fixture は既存の決定論的 helper を再利用し、ステップ定義へ製品ロジックや安全境界を複製しない。GitHub などの外部サービスには接続せず、受け入れ境界の内側をテスト用アダプターに置き換える。

## 3. 正本と MDG 記述規約

### 正本

- 実行可能な `.feature.md` だけを受け入れ仕様の正本とする。
- ファイル先頭は明示的な Feature 見出しにする。YAML/TOML フロントマターと `# language: ja` は置かない。
- 通常の Markdown 説明文は GitHub 上で読むために使えるが、Cucumber の AST では構造化された説明として保持されない。機械判定に必要な情報は Scenario、Step、Tag、または決定論的な設定へ置く。
- Feature 見出し直下には「誰に対して何を保証するか」と「なぜ必要か」を短く書く。内部関数名、fixture 名、実装手順は書かない。
- Rule は、複数シナリオを同じ業務規則でまとめる場合だけ使う。

### MDG 構文

- ファイル拡張子は `.feature.md` とする。
- Feature、Rule、Scenario、Scenario Outline、Examples は、選択中の方言のキーワードを含む Markdown 見出しにする。
- Given、When、Then、And、But は `*` を前置した箇条書きに統一する。`-` や `+` は使わない。
- Data Table と Examples の表を使う場合は GFM 表とし、Gherkin として認識されるよう 2〜5 空白で字下げする。
- Doc String を使う場合は fenced code block とする。
- タグを使う場合は `` `@tag` `` の形で対象要素の直前に置く。
- 表、Doc String、Scenario Outline は保証を明瞭にする場合だけ使い、導入のためだけには追加しない。

### シナリオの責務

- 一つのシナリオは一つの保証だけを扱う。
- シナリオ名は保証される結果を中心に書く。
- Given は利用者が理解できる事前状態、When は一つの操作または契機、Then は一つの外部観測可能な結果を表す。
- 一つのシナリオに結果ステップを一つだけ置く。Then 後の And / But、複数結果を結合した Then、独立条件をまとめた比較は禁止する。
- assertion は Then のステップ定義に直接一つだけ置く。Given、When、helper には置かない。
- fixture の作成手順をシナリオ本文に出さない。
- 関数名、fixture 名、内部専用の列挙値を本文に出さない。GitHub ラベル名、コマンド名、設定名、課題番号などの公開識別子も、操作または観測結果に影響する場合だけ明示する。
- 同じステップ文言には常に同じ業務上の意味を持たせる。意味や観測対象が違う場合は別ステップにする。
- 万能ステップを作らず、コード重複はステップ定義より下の helper で解消する。ステップ定義から別のステップ定義を呼ばない。
- Scenario Outline は入力値だけが異なり、Given / When / Then の意味と期待する保証が同じ場合だけ使う。異なる結果や業務規則は別シナリオにする。
- Data Table は一つの入力オブジェクトまたは一覧だけを表し、複数の期待結果を列へ詰め込まない。各 Example も一つの観測結果、一つの assertion とする。

日本語運用中の最小例は次の形になる。

```markdown
# 機能: 古い pull request head への push を防ぐ

レビュー中に head が変わったとき、別の変更を上書きしないために更新を停止する。

## シナリオ: 確認後に head が変わった場合は push しない

* 前提 更新対象の pull request が確認済みである
* もし push の直前に pull request head が変わる
* ならば branch への push は行われない
```

この例でも「push しない」以外のラベル、コメント、終了状態を同じ Then で検査してはならない。必要ならそれぞれ別シナリオにする。

## 4. コマンドと CI

実装後のコマンド契約は次のとおりとする。

| コマンド | 責務 |
|---|---|
| `npm run test:unit` | Vitest だけを実行する |
| `npm run test:acceptance` | Cucumber だけを実行する |
| `npm test` | `test:unit`、`test:acceptance` の順に直列実行する |
| `npm run check` | 記述規約の自動検査、`npm test`、既存の lint、型検査、shell 構文検査、package dry-run を実行する |

直列実行を初期契約とし、出力混在と fixture 競合を避ける。CI は Node.js 22 で `npm ci` の後に `npm run check` を実行し、Vitest、規約検査、Cucumber、既存の静的検査を一つも迂回しない。

実行時間の目安は Cucumber 単体でローカル 5 秒以内、Vitest と Cucumber の合計でローカル 10 秒以内とする。CI 環境の速度の揺れだけで失敗させない。目安を外れた場合はテストを減らさず、まず計測して fixture、実装、モジュール境界などの根本原因を直す。根本原因の確認後に限り、安全な並列化を検討できる。

個別実行、失敗表示、ソース位置には Cucumber 標準機能を使う。シナリオ名、`.feature.md` の行、失敗ステップ、assertion 差分、TypeScript ステップ定義の位置から原因を追える状態を保つ。

## 5. 自動検査と人間レビュー

### CI で強制する項目

Gherkin と TypeScript の構文木を読む専用 TypeScript スクリプトを `npm run check` に組み込み、Cucumber より前に実行して少なくとも次を検査する。文字列検索だけで代用しない。

- `.feature.md` にフロントマターがない。
- 各 Scenario / Scenario Outline の結果ステップが一つだけである。
- Then 後に And / But がない。
- Then のステップ定義に直接の assertion が一つだけある。
- Given / When のステップ定義と受け入れテスト用 helper に assertion がない。
- Cucumber 実行に未定義、曖昧、pending のステップがない。
- `npm run test:acceptance` で対象シナリオが一件以上検出・実行される。Cucumber-JS は言語設定を誤ると日本語 MDG を0件として終了コード0にするため、0件実行を成功として扱わない決定論的な検査を置く。
- 日本語運用中は設定が `language: 'ja'` であり、英語化完了時は `language: 'en'` である。
- 英語化完了後は対象ディレクトリに日本語 Gherkin キーワードが残っていない。

自動検査自身には、規約違反ごとに失敗理由を一つに特定できるテストを付ける。

### 人間がレビューする項目

機械判定しない次の項目を各移行 PR で確認する。

- Feature の保証対象と理由が、ステップ定義を読まずに分かるか。
- 一つのシナリオが一つの利用者観測可能な保証だけを表すか。
- Given / When / Then が事前状態、一つの契機、一つの外部結果に分かれているか。
- 内部実装名や不要な公開識別子が漏れていないか。
- 同じ概念に同じ用語を使い、ステップ再利用が業務上の意味に基づいているか。
- 既存 Vitest より失敗原因が分かりにくくなっていないか。
- fixture の独立性、実行時間、安定性を損なっていないか。
- 同等性の証拠があり、削除する Vitest が完全に置換されているか。

日本語全面移行、実運用、英語化開始、英語化完了では、後述の各ゲートについて人間の明示承認を必要とする。

## 6. 分類結果と移行順

分類の正本は [既存テストの Cucumber / Vitest / 削除候補分類](cucumber-test-classification.md) とし、この仕様へ全399件の表を転載しない。承認済みの内訳は次のとおりである。

- Cucumber 候補: 217件
- Vitest 継続: 173件
- 削除候補: 9件

Cucumber 候補は217個のシナリオへ機械変換する件数ではない。移行時に外部観測可能な保証へ書き直し、同じ保証へ統合することも、診断価値を理由に Vitest 継続へ再分類することもできる。ただし、分類表の各 ID は最終的に「移行済み」「Vitest 継続へ再分類」「同じ保証へ統合」のいずれかへ追跡できなければならない。

移行は分類表の優先グループに従い、次の順で行う。

1. **破壊操作を防ぐ安全契約**: ブランチ更新、自動修正、作業領域の片付け、プロジェクト検証、信頼確認。
2. **Issue から PR レビューまでの状態遷移**: Issue 選択、契約不足、作業開始、PR 選択、CI・外部レビュー待機、競合更新、修正再実行、古い占有の回収。
3. **人間向け表示と復旧方法**: `/deadloop-doctor`、`/deadloop-status`、停止コメント、復旧コマンド。
4. **公開設定と起動方法**: 設定の既定値・優先順位、CI 代替、エージェント起動、監視終了。

各グループ内では、分類表で同じ公開機能または安全契約を守る ID を一つの移行 PR 候補とする。PR が複数の独立した機能・安全契約を含みそうなら分割する。件数をそろえるためのバッチ化はしない。

## 7. PR ごとの実施手順

### 段階1: 土台確認 PR

最初の PR は次だけを行う。

1. Cucumber と `tsx` の依存、`cucumber.cjs`、コマンド、CI 統合を追加する。
2. 記述規約の自動検査とそのテストを追加する。
3. 優先グループ1から、安全上重要な日本語シナリオを一つ追加する。
4. 既存 Vitest は削除しない。
5. 通常の成功確認に加え、期待値を一時的に変えて Cucumber が非 0 で失敗することを確認し、元へ戻す。
6. assertion 失敗、未定義ステップ、言語設定の欠落、0件実行の拒否、TypeScript source map の位置表示を確認する。言語設定の欠落は専用の設定検査で非0終了させ、Cucumberがシナリオ0件のまま成功することを許さない。壊した状態はコミットしない。
7. PR に実行コマンド、失敗確認の要約、実行時間を記録する。
8. 人間が読みやすさ、失敗表示、実行時間を承認してからマージする。

### 段階2: 機能・安全契約ごとの移行 PR

土台確認後は、一つの機能または一つの安全契約だけを各 PR で扱い、次を同じ PR 内で完結させる。

1. 分類表から対象 ID と守る契約を列挙する。
2. 外部観測可能な一つの保証へ書き直した日本語シナリオを先に追加する。
3. 既存の決定論的 helper とテスト用アダプターを用いてステップ定義を実装する。
4. Vitest と Cucumber を併存させた状態で同等性を確認する。
5. 期待する外部結果を一時的に変え、対象 Cucumber シナリオが意図した理由で失敗することを確認して元へ戻す。
6. 完全に置換できた Vitest だけを削除する。診断価値が残るテストは Vitest 継続へ再分類する。
7. 分類 ID と、移行先シナリオ、統合先、または再分類理由の対応を PR に記録する。
8. `npm run check` を実行し、通常の PR レビューを受ける。

移行 PR に製品の動作変更や無関係な整理を混ぜない。削除候補9件は、分類表に記載された現行契約の代替テストを確認したうえで移行作業中に削除し、処理済み ID を記録する。

## 8. 同等性確認と Vitest 削除条件

同等性はテストコードの行や内部呼出しの一致ではなく、「同じ事前状態と契機に対して、同じ利用者観測可能な保証を検出できること」で判定する。各対象について次を確認する。

- 元の Vitest が守る契約と分類 ID が特定されている。
- Cucumber の Given / When が同等の事前状態と契機を作る。
- Then が元の契約の外部観測結果を一つ検査する。
- 正常系で Vitest と Cucumber がともに成功する。
- 期待結果を意図的に壊したとき、Cucumber が対象保証の失敗として非 0 終了し、位置と差分から原因を追える。
- 統合した場合は、どの分類 ID がどのシナリオで守られるか記録されている。

既存 Vitest を削除できるのは、上記を満たし、かつ次のすべてが成立する場合だけである。

- 公開保証が Cucumber で完全に置換されている。
- 小さな関数、解析、引数変換、静的ファイル検査、低レベル統合などの局所的な診断価値が残っていない。
- Cucumber の失敗原因が元のテスト以上に不明瞭になっていない。
- PR の人間レビューで削除が承認されている。

一部しか置換できない場合は元の Vitest を残すか、重複しない単位に分割して置換済み部分だけを削除する。同等性確認前に削除してはならない。

## 9. 停止条件

次のいずれかが起きた PR では既存 Vitest を削除せず、その契約の移行を停止する。

- 外部から同じ保証を表現できない。
- シナリオが内部実装の説明になる。
- 失敗原因が以前より分かりにくい。
- 実行時間が大幅に増える。
- 不安定に失敗する。

停止時は件数達成を優先しない。計測結果と原因を記録し、必要なら製品コード、fixture、モジュール境界、テスト境界を改善する別 Issue を作ってから再開する。実行時間については、承認済みの目安を超えたという理由だけでテストを削らず、根本原因を調査する。

## 10. 日本語全面移行ゲート

次のすべてを満たし、人間が明示承認した時点で日本語全面移行の完了とする。

- Cucumber 候補217件の各 ID が「移行済み」「Vitest 継続へ再分類」「同じ保証へ統合」のいずれかで処理されている。
- 削除候補9件が削除されている。
- 置換した Vitest と Cucumber シナリオの対応が追跡できる。
- 未定義、曖昧、pending のステップや保留中シナリオがない。
- `npm run check` と CI が成功する。
- 実行時間が目安内にあるか、遅さの根本原因を調査する Issue が作られている。
- 人間が分類表、対応記録、代表的なシナリオを確認し、全面移行完了を明示承認している。

このゲートまでは Cucumber 設定を `language: 'ja'` に保つ。

## 11. 実運用ゲート

日本語全面移行後、実際に必要になった仕様変更または不具合修正を一件選ぶ。架空の確認用変更は作らない。

1. 変更する保証を日本語シナリオとして先に追加または変更する。
2. そのシナリオが期待した理由で失敗することを確認する。
3. 製品コードを変更してシナリオを成功させる。
4. `npm run check` と CI を成功させ、通常の PR レビューを完了する。
5. 人間が、日本語 Cucumber を実際の変更で読み、失敗を調査し、保守できたと明示承認する。

期間の経過だけではこのゲートを通過しない。この承認後に限り英語化へ進める。

## 12. 英語化契約

### 開始条件

次の三条件をすべて満たし、人間が明示承認してから開始する。

1. 承認範囲の日本語 Cucumber 移行が完了している。
2. 実際の仕様変更または不具合修正を、日本語シナリオの失敗から成功まで一度経験している。
3. 人間がシナリオを読み、今後も保守できる書き方だと承認している。

### 実施方法

- 英語化専用の一つのブランチと PR で行い、製品動作、機能修正、保証の追加・削除を混ぜない。
- 先に用語表を作り、同じ概念を同じ英語へ対応させる。
- GitHub ラベル名、コマンド名、設定名などの公開識別子は翻訳しない。
- ステップ定義より下の helper とテスト用アダプターは変えず、日本語のシナリオとステップ表現、および対応するステップ定義を英語へ置き換える。
- 翻訳前後で同じシナリオが同じ結果になることを確認する。
- 人間が翻訳前後を比較し、保証の意味が変わっていないことを確認する。

### 完了条件

- `cucumber.cjs` の言語が `language: 'en'` になっている。
- `.feature.md` の Feature / Rule / Scenario / Step と対応するステップ定義が英語になっている。
- 日本語版 `.feature.md` や日本語ステップ定義が別名で残っていない。Git 履歴に残ることは問題としない。
- README などに日本語シナリオを転載せず、英語の正本へリンクしている。
- 自動検査が対象ディレクトリの日本語 Gherkin キーワード残存を失敗させる。
- `npm run check` と CI が成功する。
- 人間が意味の同等性と英語を唯一の正本にしたことを明示承認している。

## 13. 検証コマンド

土台 PR と各移行 PR では、変更範囲に応じた個別確認に加えて次を実行する。

```bash
npm run test:unit
npm run test:acceptance
npm test
npm run check
```

土台 PR ではさらに、一時的な変更を元へ戻すことを前提に、次の否定確認を独立に行う。

- assertion の期待値を変えると対象シナリオが失敗する。
- 未定義ステップがあると失敗する。
- `language: 'ja'` を外すと専用の設定検査が非0終了する。
- 探索先や言語設定を誤って実行対象が0件になると、`npm run test:acceptance` が非0終了する。
- TypeScript の失敗位置が source map を通じて表示される。
- `.feature.md` が設定した探索先から実際に検出・実行される。

各否定確認後に作業ツリーを確認し、意図的に壊した内容を残さない。全 PR で `git diff --check` と `git status --short` も確認する。

## 14. 完了条件

Cucumber 移行全体は、次をすべて満たしたときに完了する。

- 土台確認 PR が人間レビューを通過している。
- 日本語全面移行ゲートと実運用ゲートが明示承認されている。
- 英語化契約の完了条件がすべて満たされ、人間が承認している。
- 英語 `.feature.md` が受け入れ仕様の唯一の正本である。
- Vitest には分類上の継続対象と、移行時に診断価値を理由として再分類されたテストだけが残る。
- Cucumber と Vitest の双方を含む `npm run check` と CI が成功する。
- 停止中の移行項目、未処理の分類 ID、未定義・曖昧・pending ステップがない。
