# 作業エージェント起動・監視の Cucumber 移行記録

Issue #119 では、Issue 専用の作業場所で担当を重複なく起動し、完了ファイルが確定するまで安全に監視する保証を、日本語の実行可能な受け入れ仕様へ移した。製品動作は変更していない。

## 分類 ID と受け入れ仕様の対応

| 分類 ID | 移行先または扱い | 状態 |
|---|---|---|
| T001 | 既存 pull request の作業場所を開く reviewer 固有の起動引数、プロンプト書込み、戻り値を一度に診断する低レベル統合テスト | Vitest 継続へ再分類 |
| T002 | 「完了済みの同名担当は片付けてから交代する」 | 移行済み |
| T003 | 「稼働中の同名担当は重複起動しない」 | 移行済み |

| T005 | 「別の作業場所にいる同名担当は片付けない」 | 移行済み |

移行済みの T002〜T006、T215、T391〜T396 は、同じ製品 helper または coordinator fixture を通る受け入れシナリオで置換したため、元の Vitest を削除した。各 Then は一つの外部結果を一つの assertion で確認する。

## Vitest を残す境界

次の分類 ID は、受け入れ仕様より失敗箇所を狭く示すため Vitest に残す。

- T001: reviewer が既存の作業場所を開く場合の、実行基盤から共通ランチャーまでの低レベル統合。
- T175〜T186: Herdr コマンドの構築、応答の正規化、不正応答の拒否という実行基盤アダプター境界。
- T216〜T219、T222〜T224: 決定論的な名前と完了ファイルのパス、完了根拠を伝えるプロンプト責務、環境変数、描画処理の局所診断。
- T387〜T390: 監視プロンプトの静的文言と polling 終了記述の検査。
- T397〜T399: 時刻の解析と CLI 引数エラーの低レベル診断。

これらは公開状態遷移を重複して確認するためではなく、アダプター、文字列、解析、引数のどこが壊れたかを直接示すために残す。

## 同等性と意図的な失敗の確認

2026-07-24 に、削除前の Vitest と新しい Cucumber を併存させて `npm run test:unit` と `npm run test:acceptance` を実行し、両方が成功することを確認した。置換後は Vitest 56ファイル 744テスト、Cucumber 91シナリオ 475ステップが成功した。

同日、「準備済み Issue の担当には基準ブランチから専用の作業場所を作る」の Then で、期待する基準ブランチを一時的に `origin/intentional-failure` へ変更した。`npm run test:acceptance` は終了状態1となり、対象シナリオだけが失敗して実際値 `origin/main` と一時的な期待値の差を報告した。期待値を `origin/main` へ戻した後、受け入れテストが再び成功することを確認した。
