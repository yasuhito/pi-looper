# 試験利用者募集と発信素材の再利用計画

この文書は、Issue「[試験利用者募集と発信素材の再利用計画を決める](https://github.com/yasuhito/deadloop/issues/94)」で決めた、最初の試験利用者を探しながらdeadloopについて発信する方針を記録する。

## 目的

Pi + Herdrの基準経路を作者以外の利用者に試してもらい、公開文書だけでGitHub Issueから`ready-for-human`のPRまで完走できるかを確かめる。同時に、開発中の事実を継続的に発信し、次の試験利用者と利用例を見つける。

募集と発信に固定日程は置かない。人数、依頼先、同席の有無、観察形式は、反応と参加者の都合に合わせて作者が判断する。計画どおりの運用より、安全条件、主張の正確さ、同意の範囲を優先する。

## 最初に探す利用者

最初の対象は、次に近い個人の保守担当者または小規模開発チームである。

- GitHub Issueを小さな実装単位に分けている。
- コーディングエージェントを利用している、または試したい。
- 最終的なマージ判断は人間が行いたい。
- 破棄可能なリポジトリ、または安全に試せる小さなIssueを用意できる。

PiやHerdrの経験は参加条件にしない。経験のない利用者が前提準備で止まった事実も、導入経路を判断する材料になる。ただし、Pi + Herdrそのものの導入支援とdeadloopの導入障壁は記録上区別する。

## 募集経路

おおむね次の近い経路から始めるが、日数や人数を昇格条件として固定しない。

1. **Slackの個人チャンネル** — 開発状況と短いデモを示し、試験利用に関心のある人を募る。
2. **個別依頼** — 反応がなければ、条件に近い人へ一度だけ依頼する。断る理由は求めず、再依頼しない。
3. **部会** — 同じデモを使い、安全契約と確認したい導入障壁を短く説明する。
4. **X** — 他の経路の完了を待たず、開発中から継続的に投稿する。
5. **README** — 標準デモが公開可能になったら、安全な導入経路の近くへ置く。

この順序は作業の目安であり、直列の承認ゲートではない。紹介や反応があれば、複数の経路を並行して使ってよい。

## Xと一般向け発信

第三者の完走例ができるまでXへの投稿を待たない。deadloopはすでに公開され、開発過程も発信されているため、「一般公開」を一度だけの開始日として扱わない。

一方、発信内容の確度は観測事実に合わせる。

### 第三者完走前に発信できること

- deadloopが目指す「GitHub Issues in, reviewed PRs out.」という価値。
- Pi + Herdrを基準経路として開発していること。
- 作者の試験用リポジトリで確認したデモ。
- 現在の安全条件、`autoMerge: false`、`ready-for-human`で止まること。
- 開発上の判断、進捗、既知の制約。
- 試験利用者またはフィードバックの募集。

### 第三者完走前に発信しないこと

- 広い利用者層で実証済みであるという主張。
- 誰でも簡単に、または一定時間内にPRまで完走できるという主張。
- Pi + Herdr以外を第一級対応として扱う表現。
- 同意を得ていない所要時間、Issue / PR URL、発言、画面。
- 65〜75秒のデモを実際の処理時間と誤認させる表現。

第三者の完走後も、1件の成功を一般的な成功率として表現しない。同意された事実を、対象環境と制約付きの利用例として紹介する。

## 標準デモの再利用

[60〜90秒のdeadloop標準デモ案](prototypes/deadloop-standard-demo.md)で作る同一のGIF / MP4を、README、Slack、部会、Xへ流用する。媒体ごとに新しい映像は作らず、周囲の文面だけを変える。

- **README**: 何を入力し、どこで人間へ渡すかを示し、安全な導入手順へ接続する。
- **Slack**: 試験利用で何を確認したいか、どのような人を探しているかを添える。
- **部会**: 映像の前後に、安全契約と既知の制約を口頭で補う。
- **X**: 一つの開発判断、デモ、募集のいずれかに焦点を絞り、GitHubへリンクする。

標準デモを公開する前に、作者の破棄可能な公開リポジトリで実際の証拠を取得する。無音で見た人が「Issueからレビュー済みPRを作る」「自動マージせず人間へ渡す」「映像時間は処理時間ではない」と理解できることを確認する。

## 呼びかけの粗い案

完成した広告文ではなく、媒体ごとに残す情報の最小形を示す。

### Slack / 個別依頼

> GitHub Issueから、コーディングエージェントによる実装・検証・PR作成・レビューを経て、人間の確認待ちまで安全に回すdeadloopを作っています。Pi + Herdrの経路を、小さな試験用Issueで試してくれる方を探しています。自動マージは使いません。導入中に止まった場所を確認したいので、興味があれば声をかけてください。記録や公開への同意は参加条件ではありません。

### 部会

> 75秒の映像は処理時間ではなく、実際の試験実行から得た4つの節目の再演です。今日は機能紹介より、文書だけで導入できるか、安全に止まった理由が分かるかを確認してくれる試験利用者を探しています。

### X

> GitHub Issues in, reviewed PRs out. deadloopは、小さなIssueをコーディングエージェントへ渡し、検証とレビューを経て`ready-for-human`で止めるループを作っています。現在はPi + Herdr経路を重点的に検証中です。試してみたい方や、同じ作業に困っている方からのフィードバックを募集しています。

投稿時点の実装と一致するように毎回短く調整する。未実装のdoctor改善や未確認のWorkerを、利用可能であるかのように先取りしない。

## 試験利用で記録すること

自動テレメトリーは導入しない。[第三者導入観察票](prototypes/third-party-onboarding-observation.md)を必要に応じて使い、参加者が同意した範囲だけを手動で記録する。

親Issue #82で定めた90日間の成功指標を判断する際には、次の項目を使う。これは募集日程、人数、観察形式を固定するものではない。

- 作者以外の導入者数。
- doctor合格と最初のIssueのキュー投入まで到達したか。
- `ready-for-human`まで完走したIssue数。
- 同じ利用者が3件以上完走したか。
- 利用したAutomation host、実行基盤、Agent programの組み合わせ。
- 最初に止まった段階と、口頭補足なしで自己解決できたか。
- 安全停止の理由を理解し、復旧できたか。
- 意図しないGitHub副作用があったか。
- 公開を許可された利用例、所要時間、発言の範囲。

同席観察、非同期の報告、短い聞き取りのどれを使うかは固定しない。参加者の都合と状況に合わせる。ただし、作者の口頭補足が必要だった完走は、第一級対応の「公開文書だけで完走」の証拠には数えない。

## フィードバックから共通改善を選ぶ

個別の要望をすべてロードマップへ追加しない。次の順で共通改善の候補を判断する。

1. 危険側へ進ませた、または意図しないGitHub副作用を起こした問題。
2. 複数人を同じ段階で止めた問題。
3. 一人でも文書から自己解決できなかった問題。
4. 複数人へ同種の補足が必要だった問題。
5. 完走までの時間へ大きく影響した問題。

一度の募集波で優先する共通改善は上位1〜2件に絞る。環境固有の問題は再現条件を記録して保留し、一般化できる契約が見つかるまで専用統合を増やさない。

## 守る境界

- 最初の試験利用では`autoMerge: false`を使う。
- 記録、画面共有、公開への同意を参加条件にしない。
- APIキー、トークン、ローカルパス、非公開リポジトリ名を記録しない。
- 素材利用の同意は、製品試験への参加同意と分けて確認する。
- Claude App / Codex App内の経路を、Pi + Herdrと同じ第一級対応として発信しない。
- 独立Webサイト、Discord、自動テレメトリー、媒体別の映像制作を先に始めない。

## 次の実行項目

日付や厳密な順番ではなく、依存関係だけを残す。

- 作者の破棄可能な公開リポジトリで標準デモを収録し、理解確認を行う。
- 現在のSlack、部会、Xで同じ素材と価値提案を使う。
- 得られた参加者に合わせて観察形式を決める。
- 同意された事実を手動で集約する。
- 共通障壁の上位1〜2件を実装Issueへ昇格させる。
- 第三者完走例ができたら、条件と制約を伴う利用例としてREADMEと各発信面へ戻す。
